## 第一步：修复 TCP 粘包/拆包问题

### 问题背景

当前 `MaCacheDecoder` 的实现假设每次 `decode()` 调用都能收到完整的 RESP 消息，但 TCP 是流式协议，可能出现：

- **粘包**：多条命令在一个 TCP 包中到达（`SET foo bar\r\nGET foo\r\n`）
- **拆包**：一条命令被拆分到多个 TCP 包（`*3\r\n$3\r\nSET` | `\r\n$3\r\nfoo\r\n$3\r\nbar\r\n`）

当前实现虽然有基本的完整性检查，但逻辑不够健壮，需要改进。

### 修复方案

保持当前 `MaCacheDecoder` 的基本结构，但增强完整性检查逻辑：

1. 对于 RESP 数组格式（`*N\r\n...`），精确计算需要的行数
2. 对于 inline 格式（`PING\r\n`），等待完整的 `\r\n`
3. 使用 `markReaderIndex()` 和 `resetReaderIndex()` 实现回退机制
4. 只有确认消息完整后才消费字节并传递给 handler

### 文件变更

- 修改：`server/MaCacheDecoder.java`

## 第二步：TTL / EXPIRE 支持

### 功能说明

为 key 设置过期时间，到期后自动删除。

### 数据结构变更

修改 `CacheEntry` 类，增加过期时间字段：

```java
public class CacheEntry<T> {
    private T value;
    private long expireAt = -1;  // -1 表示永不过期，否则为毫秒时间戳

    public boolean isExpired() {
        return expireAt > 0 && System.currentTimeMillis() > expireAt;
    }
}
```

### 删除策略

#### 惰性删除

每次访问 key 时检查是否过期，过期则删除并返回 nil：

```java
// MaCache.java
private <T> CacheEntry<T> getEntry(String key) {
    CacheEntry<?> entry = map.get(key);
    if (entry != null && entry.isExpired()) {
        map.remove(key);
        return null;
    }
    return (CacheEntry<T>) entry;
}
```

所有读操作（`get`/`llen`/`scard`/`hget`/`zcard` 等）都通过 `getEntry()` 获取数据。

#### 定期删除

后台线程每 100ms 扫描一批 key（随机抽样 20 个），清理已过期的条目：

```java
// core/ExpireTask.java
@Component
public class ExpireTask {
    @Autowired private MaCache cache;

    @PostConstruct
    public void start() {
        ScheduledExecutorService executor = Executors.newSingleThreadScheduledExecutor();
        executor.scheduleAtFixedRate(() -> cache.cleanExpired(), 100, 100, TimeUnit.MILLISECONDS);
    }
}
```

### 新增命令

| 命令 | 说明 |
|------|------|
| `EXPIRE key seconds` | 设置过期时间（秒） |
| `PEXPIRE key milliseconds` | 设置过期时间（毫秒） |
| `TTL key` | 返回剩余秒数，-1 表示永不过期，-2 表示 key 不存在 |
| `PTTL key` | 返回剩余毫秒数 |
| `PERSIST key` | 移除过期时间 |

## 第三步：类型安全检查

### 问题背景

当前对错误类型操作会抛出 `ClassCastException`，导致服务器崩溃或返回不友好的错误信息。

例如：
```bash
SET mystr hello
LPUSH mystr world  # 期望返回 WRONGTYPE 错误，实际会抛异常
```

### 修复方案

在 `MaCache` 每个方法入口增加类型校验，类型不匹配时抛出自定义异常：

```java
// core/WrongTypeException.java
public class WrongTypeException extends RuntimeException {
    public WrongTypeException() {
        super("WRONGTYPE Operation against a key holding the wrong kind of value");
    }
}

// MaCache.java
private LinkedList<String> getList(String key) {
    CacheEntry<?> entry = getEntry(key);
    if (entry == null) return null;
    Object value = entry.getValue();
    if (!(value instanceof LinkedList)) {
        throw new WrongTypeException();
    }
    return (LinkedList<String>) value;
}
```

在 `MaCacheHandler` 中捕获此异常并返回标准错误响应：

```java
try {
    Reply<?> reply = command.exec(cache, args);
    ctx.writeAndFlush(encode(reply));
} catch (WrongTypeException e) {
    ctx.writeAndFlush("-WRONGTYPE Operation against a key holding the wrong kind of value\r\n");
} catch (Exception e) {
    ctx.writeAndFlush("-ERR " + e.getMessage() + "\r\n");
}
```

### 文件变更

- 新建：`core/WrongTypeException.java`
- 修改：`core/MaCache.java`（所有 `getList`/`getSet`/`getHash`/`getZset` 方法增加类型检查）
- 修改：`server/MaCacheHandler.java`（增加异常捕获）

## 第五步：验证方式

```bash
# 启动服务
mvn spring-boot:run

# 连接测试
redis-cli -p 6379

# === TTL 测试 ===
SET session_token abc123
EXPIRE session_token 5
TTL session_token
# 期望: (integer) 5（或 4/3，取决于执行速度）

# 等待 6 秒
GET session_token
# 期望: (nil)

TTL session_token
# 期望: (integer) -2  (key 不存在)

# === PERSIST 测试 ===
SET temp_key hello
EXPIRE temp_key 10
TTL temp_key
# 期望: (integer) 10（或更小）

PERSIST temp_key
TTL temp_key
# 期望: (integer) -1  (永不过期)

# === 类型安全测试 ===
SET mystr hello
LPUSH mystr world
# 期望: (error) WRONGTYPE Operation against a key holding the wrong kind of value

RPUSH mylist a b c
HSET mylist field value
# 期望: (error) WRONGTYPE Operation against a key holding the wrong kind of value

SADD myset x y z
ZADD myset 100 member
# 期望: (error) WRONGTYPE Operation against a key holding the wrong kind of value

# === TCP 粘包测试（用 nc 模拟分片发送）===
# 在两个终端分别执行，验证服务器能正确处理分片数据
printf "*3\r\n\$3\r\nSET\r\n" | nc localhost 6379 &
sleep 0.1
printf "\$5\r\nhello\r\n\$5\r\nworld\r\n" | nc localhost 6379

# 验证数据写入成功
redis-cli -p 6379 GET hello
# 期望: "world"
```
