# MaCache v1.0 Bug Fix Report

## 概述

v1.0 初始实现完成后，`redis-cli -p 6379` 无法正常连接。本文记录排查过程、根本原因及修复方案。

---

## Bug 1：端口被旧进程占用

### 现象

启动时报错：

```
java.net.BindException: Address already in use
```

### 原因

上一次启动的 MaCache 进程未被正确终止，仍占用 6379 端口。

### 排查

```bash
lsof -i :6379 | grep LISTEN
# java  54334  malinghan  114u  IPv6 ...  TCP *:6379 (LISTEN)
```

### 修复

```bash
kill -9 <PID>
```

---

## Bug 2：响应无法发送（核心 Bug）

### 现象

服务器启动正常，`redis-cli -p 6379 PING` 一直 hanging，没有任何响应返回。

### 排查

在 `MaCacheDecoder` 和 `MaCacheHandler` 中加入日志后确认：

- 数据确实到达了 decoder
- 数据也传递到了 handler
- `command.exec()` 被正确调用
- 但客户端始终收不到任何字节

```
Decoder received: *1\r\n$4\r\nPING\r\n
Received: *1\r\n$4\r\nPING\r\n
```

### 根本原因

Netty pipeline 中缺少 `StringEncoder`。

`MaCacheHandler` 调用 `ctx.writeAndFlush(String)` 写出的是 Java `String` 对象，但 Netty 不知道如何将其序列化为字节流。没有 `StringEncoder` 的情况下，出站数据被静默丢弃，客户端永远收不到响应。

### 修复

在 `MaCacheServer.initChannel()` 中添加 `StringEncoder`：

```java
// MaCacheServer.java - 修复前
pipeline.addLast(new MaCacheDecoder());
pipeline.addLast(handler);

// MaCacheServer.java - 修复后
pipeline.addLast(new MaCacheDecoder());
pipeline.addLast(new StringEncoder(CharsetUtil.UTF_8));  // ← 新增
pipeline.addLast(handler);
```

## Bug 3：redis-cli 交互模式无法连接

### 现象（第一阶段）

`redis-cli -p 6379 PING` 可以正常工作，但 `redis-cli -p 6379`（交互模式）连接后立即断开，提示：

```
HELLO 3 failed: ERR unknown command 'HELLO'
```

### 原因

`redis-cli` 进入交互模式时，会先发送 `HELLO 3` 命令进行 RESP3 协议握手。服务器返回 unknown command 错误后，redis-cli 判定服务器不兼容而断开连接。

### 修复（第一阶段）

新增 `HelloCommand`，返回兼容 RESP2 的握手响应：

```java
// command/HelloCommand.java
public class HelloCommand implements Command {
    @Override
    public String name() { return "HELLO"; }

    @Override
    public Reply<?> exec(MaCache cache, String[] args) {
        return Reply.array(Arrays.asList(
            "server", "redis",
            "version", "7.0.0",
            "proto", "2",
            "mode", "standalone",
            "role", "master"
        ));
    }
}
```

并在 `Commands.java` 中注册：

```java
register(new HelloCommand());
```

### 现象（第二阶段）

第一阶段修复后，`redis-cli -p 6379` 仍然崩溃，报错：

```
Assertion failed: (commandTable->element[i + 1]->type == REDIS_REPLY_MAP ||
commandTable->element[i + 1]->type == REDIS_REPLY_ARRAY),
function cliCountCommands, file redis-cli.c, line 680.
```

### 根本原因

第一阶段的修复存在致命缺陷：

1. redis-cli 发送 `HELLO 3` 请求升级到 RESP3 协议
2. 服务器错误地返回了成功响应（包含 `proto 2`）
3. redis-cli 误以为协议升级成功，后续按 RESP3 模式运行
4. redis-cli 发送 `COMMAND DOCS` 并期望返回 **map 类型**（RESP3 特性）
5. 服务器返回的是 RESP2 的 array，触发 redis-cli 内部断言失败

**协议不一致的根源**：我们返回了 `proto 2`，但 redis-cli 认为这是对 `HELLO 3` 的成功响应，导致客户端和服务器对当前协议版本的理解不一致。

### 修复（第二阶段 - 最终方案）

**正确做法**：直接拒绝 `HELLO 3` 请求，让 redis-cli 自动降级到 RESP2 模式。

```java
// command/HelloCommand.java - 最终版本
public class HelloCommand implements Command {
    @Override
    public String name() { return "HELLO"; }

    @Override
    public Reply<?> exec(MaCache cache, String[] args) {
        // 拒绝 RESP3 协议升级请求
        if (args.length > 1 && "3".equals(args[1])) {
            return Reply.error("NOPROTO unsupported protocol version");
        }
        // 仅支持 RESP2
        return Reply.array(Arrays.asList(
            "server", "redis",
            "version", "7.0.0",
            "proto", "2",
            "mode", "standalone",
            "role", "master"
        ));
    }
}
```

**工作流程**：

1. redis-cli 发送 `HELLO 3`
2. 服务器返回 `-NOPROTO unsupported protocol version\r\n`
3. redis-cli 收到错误后自动降级，发送 `HELLO 2`（或不带参数的 `HELLO`）
4. 服务器返回 RESP2 握手响应
5. redis-cli 以 RESP2 模式运行，不再发送 `COMMAND DOCS` 等 RESP3 专属命令
6. 交互模式正常工作

### 关键教训

**不要对不支持的协议版本返回成功响应**。即使响应体里声明了 `proto 2`，客户端仍会按照请求的版本（RESP3）来解析后续通信，导致协议不匹配。

正确的协议协商流程：
- 客户端请求不支持的版本 → 服务器返回错误 → 客户端降级重试
- 客户端请求支持的版本 → 服务器返回成功 → 双方按约定版本通信

## Bug 4：Command 接口参数索引设计错误

### 现象

Bug 2 修复后，`SET`/`GET` 等命令执行时会抛出 `ArrayIndexOutOfBoundsException`。

### 原因

初始设计中，`Command` 接口的 `getKey()`/`getVal()`/`getParams()` 假设 `args[]` 是 RESP 原始分割数组（包含 `*N`、`$len` 等元数据行）：

```java
// 原始设计（错误）
// 假设 args = ["*3", "$3", "SET", "$3", "foo", "$3", "bar"]
default String getKey(String[] args) { return args[4]; }  // "foo"
default String getVal(String[] args) { return args[6]; }  // "bar"
```

但 `MaCacheHandler` 在解析时已经过滤掉了所有元数据行，传给 `exec()` 的是纯净的值数组：

```java
// 实际传入（正确）
// args = ["SET", "foo", "bar"]
```

两者不一致导致索引越界。

### 修复

重写 `Command` 接口的默认方法，匹配实际传入的纯净数组格式：

```java
// Command.java - 修复后
// args = [cmd, key, value, ...]
default String getKey(String[] args) {
    return args.length > 1 ? args[1] : null;
}

default String getVal(String[] args) {
    return args.length > 2 ? args[2] : null;
}

default String[] getParams(String[] args) {
    if (args.length <= 1) return new String[0];
    String[] params = new String[args.length - 1];
    System.arraycopy(args, 1, params, 0, args.length - 1);
    return params;
}
```

## Bug 5：MaCacheDecoder 未等待完整消息

### 现象

TCP 分包场景下，decoder 可能只收到部分 RESP 数据就立即 decode，导致 `args` 数组不完整，命令解析失败。

### 原因

原始实现每次收到任意字节就立即处理，没有检查消息完整性：

```java
// 原始实现（有风险）
protected void decode(...) {
    byte[] bytes = new byte[in.readableBytes()];
    in.readBytes(bytes);
    out.add(new String(bytes, StandardCharsets.UTF_8));
}
```

### 修复

在 decoder 中加入完整性检查：RESP 数组格式需要收到 `*N` 指定数量的完整行才处理，否则重置 readerIndex 等待更多数据：

```java
protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
    in.markReaderIndex();
    String msg = in.toString(StandardCharsets.UTF_8);

    if (msg.startsWith("*")) {
        String[] lines = msg.split("\\r\\n", -1);
        int n = Integer.parseInt(lines[0].substring(1));
        int needed = 1 + n * 2;
        if (lines.length < needed + 1) {
            in.resetReaderIndex();  // 数据不完整，等待
            return;
        }
    } else {
        if (!msg.contains("\r\n")) {
            in.resetReaderIndex();  // inline 命令未结束，等待
            return;
        }
    }

    in.readBytes(in.readableBytes());
    out.add(msg);
}
```
